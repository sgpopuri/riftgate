# v0.5 Retrospective

**Timeline**: v0.5 implementation cycle, completed 2026-07-04 in the same documentation-first execution flow as prior milestones.
**Milestone focus**: MCP capability broker — first-class agentic extension plane support with per-tenant allowlists, WAL-backed audit, and HMAC attestation.
**Close-out posture**: v0.5 is complete and closed. All ADR 0015 compliance items are satisfied.

---

## What Shipped

### ADR and Design Surface

- ADR 0015 accepted (2026-07-04); status changed from `proposed` to `accepted`.
- `CapabilityBroker` trait in `crates/riftgate-core/src/capability.rs` updated from the v0.1 placeholder (`check_tool` / `check_resource`) to the ADR 0015 `authorize(request, identity) -> CapabilityDecision` signature.
- Full v0.5 type system landed in `riftgate-core`: `McpRequest`, `LifecycleMethod`, `TenantIdentity`, `CapabilityDecision`, `AttestationHeaders`, `HmacSignature`, `DenialReason`, `McpAuditEvent`, `AuditDecision`.

### `crates/riftgate-mcp`

New crate. Implements the gateway-as-broker posture end-to-end:

- **Parser** (`parser.rs`): JSON-RPC 2.0 envelope parsing; typed `McpRequest` emission for all seven MCP method families (`tools/call`, `tools/list`, `resources/read`, `resources/list`, `prompts/get`, `prompts/list`, protocol lifecycle). SHA-256 argument hashing for audit PII-safety.
- **`AllowlistBroker`** (`allowlist.rs`): per-tenant composition of three data structures per the LLD — `HashSet<String>` for tool allow/deny (O(1) lookup), sorted `Vec<String>` with binary-search prefix matching for resource URIs (O(log n + prefix_len)), and sorted `Vec<(SystemTime, String)>` with partition-point skip for time-bounded grants (O(log n)). All three structures compiled at construction time.
- **`DryRunBroker`** (`dryrun.rs`): generic wrapper over any `CapabilityBroker`; logs would-be deny reasons via `tracing::info!` but always returns `Allow` (with zero-signature attestation to signal dry-run mode to downstream servers).
- **Attestation** (`attestation.rs`): HMAC-SHA256 signing via the RustCrypto stack (`hmac + sha2`). `sign()` produces the four attestation headers (`riftgate-mcp-caller`, `riftgate-mcp-tool`, `riftgate-mcp-decision`, `riftgate-mcp-signature`). `verify()` is available for downstream server harnesses and integration tests.
- **Audit** (`audit.rs`): serializes `McpAuditEvent` as newline-delimited JSON and appends via `WAL::append` with `Durability::Fsync`. Inline hex encoder (no external crate). Every allow and deny decision is recorded.
- **Tests**: 22 unit tests (parser, allowlist, attestation, audit, dryrun) + 3 integration tests (`wal_audit.rs`): every `authorize()` call produces one WAL record; records are valid NDJSON with correct decisions; allow decisions carry valid non-zero attestation signatures.

### Config Schema (`riftgate-config`)

New TOML surface under `[mcp]`:

```toml
[mcp]
enforce          = true          # false = DryRunBroker
signing_key_hex  = "<64 hex>"   # HMAC key; ephemeral random if absent
wal_path         = "/var/riftgate/mcp-audit"  # FileWal path; NoopWal if absent

[mcp.tenants.acme]              # human-readable name OR numeric id
allowed_tools             = ["search-web", "read-file"]
denied_tools              = ["filesystem-write"]
allowed_resource_prefixes = ["s3://acme-datasets/*"]
time_bounded_grants = [
  { tool = "send-email", until_unix_secs = 1780000000 },
]
```

### Binary Wiring (`crates/riftgate`)

- `crates/riftgate/src/mcp.rs` — new module: `build_mcp_broker(&config)` constructs `AllowlistBroker` or `DryRunBroker`, wires `FileWal` when `mcp.wal_path` is set and `NoopWal` otherwise.
- **Tenant name hashing**: non-numeric tenant keys (e.g. `"acme"`) are mapped to `u32` `TenantId` via FNV-1a 32-bit hash for a deterministic, stable identity without a separate name-to-id table.
- `HandlerState.mcp_broker`: `Option<Arc<dyn CapabilityBroker>>` added; `None` when no tenants are configured.
- `proxy.rs` `handle()`: POST `/mcp` authorization path — parses body, resolves tenant from `x-riftgate-tenant` header (defaults to `TenantId(0)`), calls `broker.authorize()`, returns `403` with `riftgate-mcp-reason` on deny, or forwards with four attestation headers injected on allow.

## What Went Well

**ADR-first discipline maintained.** ADR 0015 was written as `proposed` in v0.0, held in queue through v0.4, and formally accepted at the open of v0.5 as planned. Every line of `riftgate-mcp` code traces back to it. The process held.

**Trait abstraction cleanly absorbed the change.** Updating `CapabilityBroker` from `check_tool / check_resource` (v0.1 placeholder) to `authorize(McpRequest, TenantIdentity)` (v0.5 production) required no changes to callers outside the broker — the trait boundary held its contract. Future `ExternalPolicyBroker` (OPA/Cedar) can land behind the same trait.

**Lima as dev environment.** Replacing the implicit "some Ubuntu VM" harness with a tracked `lima/riftgate.yaml` file eliminated the setup ambiguity. Any contributor on macOS can run `limactl start lima/riftgate.yaml` and have the exact environment that CI assumes. This is the most operationally useful improvement in v0.5 beyond the broker itself.

**Test isolation layers.** Three WAL implementations — `NoopWal` (unit tests), `MemWal` (integration tests), `FileWal` (production) — kept each test layer fast and deterministic. The integration tests exercise real JSON serialization and real WAL contract assertions without requiring a real file.

**Configuration ergonomics.** FNV-1a hashing for non-numeric tenant keys (`[mcp.tenants.acme]`) means operators write natural TOML without a separate name-to-id mapping table. Deterministic hashing ensures the same key always maps to the same `TenantId` across process restarts.

## What Was Deferred (Explicitly Non-Blocking for v0.5)

**Per-request tenant resolution from config.** The current proxy path uses the raw `x-riftgate-tenant` header value as a `u32` (with `TenantId(0)` fallback). A full tenant resolution step — reading the tenant name from a JWT claim or API key registry, looking it up in config, mapping to a `TenantId` — is deferred to a v0.5 follow-on or v1.0.

**External policy engine delegation.** `ExternalPolicyBroker` (OPA, Cedar over gRPC) is per ADR 0015 a future `impl` behind a feature flag. It requires a new ADR before any code lands.

**MCP mediator posture.** Rewriting `tools/list` per tenant, substituting resource URIs — the mediator posture from Options 026 §3.4 — is explicitly deferred to v1.0+.

**Aya BPF source builds.** The `riftgate-obs-bpf-programs` crate remains environment-gated (nightly + `bpfel-unknown-none`) as carried forward from v0.4. The Lima VM environment unblocks this but the actual Aya program implementation remains in flight.

## Lessons Learned

**The "no code without a corresponding Options doc and ADR" invariant works.** ADR 0015 was queued from v0.0. Because it existed before implementation started, there was no ambiguity about what to build: the `authorize()` signature, the bit-set + prefix-trie + interval-tree composition, the WAL-backed audit, and the attestation headers were all specified before a line of implementation was written.

**`deny_unknown_fields` on config structs is an honest API contract.** New fields are additive (`#[serde(default)]`), existing fields are strict, and operators see clear error messages when they typo a field name. This is the right default for production config parsing.

**Separating config types from implementation types (config crate vs implementation crate) is the right call.** `riftgate-config` defines `McpTenantConfig`; `riftgate-mcp` defines `TenantAllowlist`. The binary converts between them at startup. This keeps both crates independent and avoids a circular dependency that would have emerged if `riftgate-config` depended on `riftgate-mcp`.

**The test coverage contract from ADR 0015 was met exactly.** Protocol-fidelity tests (parser), WAL audit round-trip test (`wal_audit.rs`), attestation verification test — all three compliance requirements from the ADR Compliance section exist in the codebase.

## Open Questions at v0.5 Close

- **HMAC key rotation.** Static key per gateway process. Rotation requires a restart. A `SigningKeyRotator` backed by a watched config file is the clean v1.0 path; document the restart procedure in RUNBOOK.md as a stop-gap.
- **Cross-replica policy coherence.** Per-instance allowlist, same as the rate limiter (ADR 0009). Operators running multiple replicas must deploy config changes in coordination. This is documented as a known limitation, not a bug.
- **MCP protocol evolution.** The parser carries an upgrade tax. When MCP revisions land, the parser is the first thing to update. Pinning the Anthropic MCP SDK version in `deny.toml` (once a Rust SDK exists) is the right mitigation.
- **Tenant id collision probability.** FNV-1a 32-bit hash has 1/2^32 collision probability per distinct name pair. For v0.5 tenant counts (tens, not millions) this is negligible. Document that operators should use numeric keys if they operate near collision risk.
