- **ADR `0009` promoted** to `accepted` at v0.2 open ([`0009-rate-limiter-trait-in-proc-only`](06-adrs/0009-rate-limiter-trait-in-proc-only.md)).
- **Four LLDs refreshed** to reflect the v0.2 trait surfaces and impls: [`lld-storage`](04-design/lld-storage.md), [`lld-routing`](04-design/lld-routing.md), [`lld-rate-limiter`](04-design/lld-rate-limiter.md), [`lld-scheduling`](04-design/lld-scheduling.md).
- **New `MpmcQueue` and `ShardedMpmcQueue`** with a `PerShardScheduler` impl in `crates/riftgate` over `crossbeam-channel`. The v0.1 binary continues to drive the accept loop on tokio per [ADR `0003`](06-adrs/0003-tokio-multithread-default.md); the scheduler types now exist behind the same `Queue<T>` + `Scheduler` traits in `riftgate-core`, ready for the binary to switch over in v0.3.
- **`TokenBucketLimiter`** in [`crates/riftgate-core`](../crates/riftgate-core/src/rate_limit.rs) — 64-shard `DashMap`, packed `AtomicU64` (token mantissa + last-refill epoch ms, `SCALE = 1 << 16`), CAS refill loop, denial vocab via [`DenialReason`](../crates/riftgate-core/src/backpressure.rs). Per [ADR `0018`](06-adrs/0018-token-bucket-parameters.md). `NoopLimiter` ships alongside it as the deterministic test/opt-out impl.
- **`WeightedRandomRouter`** in [`crates/riftgate-router`](../crates/riftgate-router/src/weighted.rs) — Vose alias method (1991), inline xorshift64* + splitmix64 (no `rand` dep), `MAX_WEIGHTED_BACKENDS = 32`, `with_seed` for deterministic tests, statistical fairness test under multi-threaded contention.
- **`CircuitBreakerArbiter<R>`** in [`crates/riftgate-router`](../crates/riftgate-router/src/circuit.rs) — three-state decorator over any `Router`. Closed → Open on N consecutive 5xx/408/429, Open → HalfOpen after cooldown, HalfOpen → Closed on a success or back to Open on the first failure. State packed in `AtomicU64`, transition timestamps in `Mutex<HashMap<BackendId, Instant>>`.
- **`BackpressurePolicy` trait + `HighWaterPolicy`** in [`crates/riftgate-core`](../crates/riftgate-core/src/backpressure.rs) — drop-newest 503 with hysteresis (separate low/high water marks, atomic shedding flag). `DenialReason::{RateLimit, QueueFull, CircuitOpen}` is the shared vocab across the three denial paths.
- **`crates/riftgate-replay`** — `FileWal` impl of `riftgate_core::wal::WAL` per [ADR `0013`](06-adrs/0013-append-only-file-wal.md). Per-shard segment files named `seg-{shard:04}-{seqno:020}.wal`, length-prefixed framing (`[u32 length][u8 durability_tag][u8 entry_kind][u64 timestamp_nanos][u64 entry_id][payload bytes]`), one group-commit `fdatasync` flusher thread per shard, mixed-durability append (`Async` returns immediately, `FdataSync`/`Fsync` park on a condvar until the flusher reports durability), segment rollover at a configurable byte ceiling, graceful shutdown that drains the flusher.
- **`crates/riftgate-io-uring`** — second concrete `AsyncIO` impl, gated by the `io-uring` Cargo feature and `cfg(target_os = "linux")`. On every other target — and on Linux without the feature — the crate compiles to an empty library so the workspace builds on macOS/Windows hosts. v0.2 ships the scaffold (construction, `register`/`deregister`/`poll` plumbing against `IORING_OP_POLL_ADD`); the shared conformance harness lifts from `riftgate-io-epoll/tests/conformance.rs` in v0.3.
- **`benchmarks/v0.2-headline/`** — the harness contract for the v0.3 perf-stabilization sweep. No headline numbers ship in v0.2 (see Open questions); the directory anchors the reproducibility commitment.
- **Documentation sync.** This retrospective + the "Currently shipping" block in [`docs/02-mvp-roadmap.md`](02-mvp-roadmap.md) carry the v0.2 → v0.3 flip.

## What went well

- **The trait + dual-impl discipline survived its second milestone.** Every v0.2 subsystem ships at least two impls behind its trait: `TokenBucketLimiter` + `NoopLimiter`; `WeightedRandomRouter` + `CircuitBreakerArbiter` decorating an arbitrary `Router`; `FileWal` + the `WAL` trait surface that downstream tests substitute. The pluggability pillar (AGENTS.md §9, "Programmable Rust core") held without compromise.
- **Packed `AtomicU64` state worked twice.** Both `TokenBucketLimiter` (tokens mantissa + epoch ms) and `CircuitBreakerArbiter` (state tag + consecutive failure count) use the same pattern: a single `AtomicU64` with bit-packed fields and a CAS retry loop. The pattern is now a documented v0.2 idiom — see the inline comments in [`rate_limit.rs`](../crates/riftgate-core/src/rate_limit.rs) and [`circuit.rs`](../crates/riftgate-router/src/circuit.rs).
- **Vose's alias method earned its weight.** O(1) weighted sampling without per-call CDF traversal lets `WeightedRandomRouter::pick` stay in the hot path. The 32-backend ceiling is generous for v0.2 use cases and the assert is loud enough that anyone hitting it will notice during the first integration test.
- **Empty-on-non-Linux crate compile worked cleanly.** Putting `io-uring` under `target.'cfg(target_os = "linux")'.dependencies` with `optional = true`, and gating every module on `cfg(all(target_os = "linux", feature = "io-uring"))`, means the workspace builds on macOS hosts without the `io-uring` crate ever entering the dependency graph. The pattern is now reusable for any Linux-only future backend.
- **`BackpressurePolicy` hysteresis matched the LLD.** The atomic shedding flag flips from `false` to `true` when queue depth crosses the high-water mark and only flips back at the low-water mark. The asserts at construction time (`0 < low < high`) are a deliberate "fail loudly" choice (per the v0.1 config-validation pattern) and caught one bench misconfiguration during local testing.

## What we missed (and fixed in the close-out PR)

This retrospective is the surface that records what we did not catch during the milestone itself. The following items were surfaced during the close-out audit and corrected before opening `v0.3`:

- **`MpmcQueue<T>` generics leaked `T: Send` through the trait impl.** First cut defined `push`/`pop`/`len` only inside the `Queue<T> for MpmcQueue<T>` impl gated on `T: Send`, but `ShardedMpmcQueue::pop_from` needed to call them on a generic `T`. Fixed by adding inherent `push_inner`/`pop_inner`/`len_inner` methods and delegating from the trait impl. Documented in the module doc-comment so the next contributor doesn't re-discover the constraint.
- **`CircuitBreakerArbiter` clippy `collapsible_match`.** A `(TAG_CLOSED, false)` match arm wrapped an `if failures > 0` predicate; clippy correctly flagged it as inlinable. Fixed by collapsing the predicate into the match guard `(TAG_CLOSED, false) if failures > 0 =>`.
- **`riftgate-io-uring` workspace `unsafe_code = "deny"` collision.** The `io-uring` crate's `SubmissionQueue::push` is an `unsafe` API by design (entries must outlive kernel consumption). Resolved by overriding `unsafe_code = "allow"` locally in [`crates/riftgate-io-uring/Cargo.toml`](../crates/riftgate-io-uring/Cargo.toml) with a justifying comment in the submission site, matching the override pattern documented in the workspace `Cargo.toml`.
- **`FileWalConfig` struct-update syntax tripped clippy.** Initial tests used `..FileWalConfig::default()` after specifying every field; `clippy::needless_update` correctly flagged it. Fixed by removing the redundant tail. (Lesson: don't reach for `..Default::default()` in a struct literal that already exhausts the field list.)

## Open at `v0.2` close

Items that are explicitly known and tracked but not blockers:

- **No v0.2 headline benchmarks.** Per AGENTS.md §5, we ship numbers only when they're reproducible from the repo against a real baseline. v0.2 delivered the subsystems that the benches need (`TokenBucketLimiter`, `WeightedRandomRouter`, `CircuitBreakerArbiter`, `FileWal`, `MpmcQueue`); v0.3 ships the bench targets in `benchmarks/v0.2-headline/` and runs the LiteLLM/TensorZero comparisons. The directory exists at v0.2 close to anchor the contract.
- **Conformance harness lift.** `crates/riftgate-io-epoll/tests/conformance.rs` is still epoll-private. The lift into a shared `riftgate-core` `tests/io_conformance/` module that both backends import is v0.3 work, gated on having Linux CI to actually exercise the `io_uring` path.
- **`PerShardScheduler` binary cutover.** The `MpmcQueue` + `PerShardScheduler` types ship in v0.2 behind the same trait, but the production binary in `crates/riftgate` still uses the tokio multi-thread runtime per [ADR `0003`](06-adrs/0003-tokio-multithread-default.md). The migration window — opt-in via `--features per-core-scheduler` — slips to v0.3 alongside the perf sweep so both can be benched together.
- **Options `022` (priority / fairness scheduling) — deferred to v0.3 retro.** Originally floated for the v0.2 close as a possible workstream; deferred because the basic three-state circuit breaker + drop-newest backpressure pair already covers v0.2's denial requirements. v0.3 retro decides whether priority queueing earns its own Options doc.
- **Open questions inherited from `v0.1`:**
  - **`BumpArena` pooling shape.** v0.2 still allocates a fresh arena per request. Production load-profile measurement carries forward to v0.3.
  - **`HierarchicalWheel` cutover criteria.** [ADR `0010`](06-adrs/0010-binary-heap-timers-v01-hierarchical-wheel-v02.md) commits the wheel to v0.2, but it did not land in this milestone — the binary-heap impl is still adequate at v0.2's scale. The cutover threshold (schedule/cancel/tick ops/sec above which the heap is no longer acceptable) is a v0.3 perf-sweep decision.
  - **Domain reservation** (`riftgate.io` / `riftgate.dev` / `riftgate.com`) — still not reserved. Recommended pre-broader-announcement decision item, now two milestones overdue.
  - **eBPF library choice** (Options `014`: bpftrace vs libbpf vs Aya) — explicitly v0.4 work; no v0.2 blocker.

## Process notes for `v0.3` and beyond

- **The per-milestone retrospective discipline holds.** This file is `v0.2`'s; `v0.3`'s will be `02d-v0.3-retrospective.md`. Retrospectives are permanent, never edited.
- **The "Currently shipping" block at the top of [`docs/02-mvp-roadmap.md`](02-mvp-roadmap.md) flips to `v0.3` at the same time this file is committed.**
- **Distribution remains GitHub + `cargo build` through v0.4.** Do not add `cargo publish` to v0.3 close-out checklists.
- **Packed-`AtomicU64`-with-CAS-loop is now an idiom.** Two v0.2 subsystems use it; document it in the next refresh of the relevant LLDs (`lld-rate-limiter`, `lld-routing`) so it stops being a per-implementation surprise.
- **Trait + dual-impl discipline (FR-X02) survives v0.2.** Hold the line for `v0.3` even when the perf-sweep tempts us to specialize a single hot-path impl.
