## What went well

- **The trait surfaces designed in `v0.0` survived implementation almost unchanged.** Two design adjustments did happen — `AsyncIO` taking `&mut self` (per-shard ownership becomes load-bearing once you hold a `mio::Poll`) and `Allocator::reset` taking `&mut self` (`bumpalo::Bump::reset` requires it) — but both were already documented as risks in the LLDs and surfaced in the first week of Phase C/D. The `Send + Sync` bounds also fell off `AsyncIO` and `TimerSubsystem` for the same per-shard-ownership reason; the `Router` and `ObservabilitySink` traits kept theirs because they actually are shared. Nothing else moved.
- **Phase ordering held.** The plan ran nine implementation phases (B → I) in dependency order — workspace bootstrap → core traits → IO → parser → config → router → obs → binary — and each phase compiled and tested green before the next phase began. The dependency-bottom-up shape meant the binary integration in Phase I was almost mechanical: every dependency was already exercised by its own tests.
- **The walking-skeleton scope was the right scope.** Tokio + hyper for the actual HTTP server and client (per ADRs `0003` / `0007`) let us ship a real, end-to-end OpenAI-shaped proxy without writing a custom HTTP server in v0.1 — while still having `riftgate-io-epoll` and `riftgate-parser` shipped as alternative impls behind their traits. The `SseFramer` *is* exercised on the upstream response body for token observation, so it isn't theoretical.
- **The drop-on-full observability bus paid off in the first week.** The contract — try-send, drop on full, count the drop — meant the data plane never blocks on observability, even when the OTel collector is down. The `request.completed` span being fired from `PinnedDrop` on the streamed-response body (rather than from `poll_frame`) caught a real bug class: spans were leaking when a client disconnected mid-stream.
- **The `examples/01-basic-openai-proxy` demo is the documentation pillar paying for itself.** A reader who clones the repo, runs `cargo run -p riftgate`, and `curl`s `/v1/chat/completions` against a mock backend can exercise every FR-001..FR-008 acceptance criterion without reading any code. The OTel collector docker-compose makes `request.received → request.routed → request.first_token → request.completed` visible in a terminal in under 60 seconds.

## What we missed (and fixed in the close-out PR)

This retrospective is the surface that records what we did not catch during the milestone itself. The following items were surfaced during the close-out audit and corrected before opening `v0.2`:

- **The end-to-end criterion bench had a silent listener-close bug.** `crates/riftgate/benches/end_to_end.rs` constructed a `(drain_tx, drain_rx)` pair, bound `drain_tx` as `_drain_tx`, then returned. `_drain_tx` was dropped at function exit (Rust binds `_name` like any other binding — only `let _ = ...` drops immediately), which made the spawned `accept_loop` see `drain.changed()` return `Err` (sender dropped) and treat it as a drain initiation. The listener was closed before the warmup TCP connect could land. Fixed by `Box::leak`-ing the sender for the lifetime of the bench process, mirroring the pattern already used for the observability `Bus`. Caught only because `cargo test --workspace --all-targets` runs benches in their criterion test mode and the warmup phase is wired to assert.
- **LLD bench paths pointed at the wrong directories.** Several v0.1 LLDs cited microbenchmarks at `crates/riftgate/benches/<name>.rs`, but the benches actually ship per-crate (`crates/riftgate-core/benches/timers.rs`, `crates/riftgate-parser/benches/http1.rs`, etc.) so each crate's `cargo bench` runs in isolation. Fixed in this close-out by walking each LLD's path references against the filesystem.
- **The `RateLimiter` and `CapabilityBroker` LLD status lines drifted.** Both were marked "outline-stage" in v0.0; v0.1 ships the trait surfaces in `riftgate-core` (no production impls) and the LLDs now name that explicitly with a forward reference to the milestone the impl lands in (v0.2 for `TokenBucketLimiter`, v0.5 for `LocalAllowlistBroker`). The `WAL` LLD got the same treatment.
- **README count drift.** The repo root README still said `v0.1` was "the next milestone" and that there was "no Rust code yet". Fixed: it now says `v0.1` walking-skeleton implementation is complete with close-out (retrospective, tag, publishing) in flight, lists the shipped subsystem table with crate locations, and points at `examples/01-basic-openai-proxy` as the entry point. AGENTS.md §2 was updated similarly.
- **`benchmarks/README.md` and `examples/README.md` still claimed they were empty during the v0.0 design phase.** Both were updated to list the v0.1 shipped artifacts inline; the planned-future tables remain for v0.2+.

## Open at `v0.1` close

Items that are explicitly known and tracked but not blockers:

- **Domain reservation.** `riftgate.io` / `riftgate.dev` / `riftgate.com` still not reserved. Carried forward from the v0.0 retrospective; recommended pre-broader-announcement decision item.
- **`PerCoreScheduler` migration window.** The v0.1 binary uses tokio multi-thread runtime per [ADR `0003`](06-adrs/0003-tokio-multithread-default.md); the custom `PerCoreScheduler` lands in v0.2 behind the same trait. The migration window is part of the v0.2 plan; the LLD at [`scheduling`](04-design/lld-scheduling.md) documents the v0.1 / v0.2 split explicitly.
- **`HierarchicalWheel` cutover criteria.** [ADR `0010`](06-adrs/0010-binary-heap-timers-v01-hierarchical-wheel-v02.md) commits to the wheel landing in v0.2; the criterion benchmark at [`riftgate-core/benches/timers.rs`](../crates/riftgate-core/benches/timers.rs) is the gate. v0.2 plan should name a target schedule/cancel/tick number above which `BinaryHeapTimers` is no longer acceptable.
- **eBPF library choice.** Options `014` (bpftrace vs libbpf vs Aya) — sequencing TBD; explicitly v0.4 work, no v0.1 blocker. Carried forward.
- **`v0.2` open questions inherited from `v0.1`:**
  - Whether the `BumpArena` pool is per-shard or shared. v0.1 allocates a fresh arena per request; pooling lands in v0.2 once we have measured allocator pressure under sustained load.
  - Whether to keep `tokio::time::timeout` in the proxy handler or migrate to the per-shard `BinaryHeapTimers` immediately when `PerCoreScheduler` lands. Documented in [`scheduling`](04-design/lld-scheduling.md).

## Process notes for `v0.2` and beyond

- **The retrospective discipline is now per-milestone.** This file is `v0.1`'s; `v0.2`'s will be `02c-v0.2-retrospective.md`, and so on. The retrospective is a permanent artifact: written once, never edited.
- **The "Currently shipping" block at the top of [`docs/02-mvp-roadmap.md`](02-mvp-roadmap.md) flips to v0.2 prep at the same time this file is committed.** That block tracks what is in flight; close-out narrative lives here.
- **Bench-as-test gating is a v0.1 lesson.** Running `cargo test --workspace --all-targets` runs criterion benches in their once-only test mode; that single pass caught a real bench-only listener-close bug that would have silently broken the bench. Keep the convention; benches must be runnable as tests.
- **Per-crate bench layout is the right shape.** Splitting benches across crates (rather than consolidating under `crates/riftgate/benches/`) means `cargo bench -p riftgate-core` can run the timer and allocator benches without compiling the entire dependency graph. Document this convention in `benchmarks/README.md` and keep it.
- **The trait + dual-impl discipline (FR-X02) survives.** Every v0.1 trait has at least two implementations: the production impl (`MioIO`, `Http1Parser`, `BumpArena`, `BinaryHeapTimers`, `RoundRobinRouter`, `OtelSink`) and a deterministic / test impl (`InMemorySink`, `ConstantRouter`, `DeterministicTimers`, `SystemAllocator`). v0.2 must hold this line as the production scheduler, file WAL, and in-proc rate limiter land.
