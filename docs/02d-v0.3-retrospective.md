## What We Missed and Fixed

### Scheduler Dispatch Model (Handle::block_on → Handle::spawn)
- **Issue**: Per-core-scheduler feature mode hung on e2e tests when worker threads called `Handle::block_on` on connection futures
- **Root Cause**: Tokio runtime has limited worker threads; blocking one prevents others from progressing
- **Fix Applied**: Changed to `Handle::spawn` for futures submitted to scheduler workers
- **Lesson**: Feature-mode E2E tests validated the fix and prevented regressions

### WasmFilter API Adjustments (3 iterations)
1. **Bindgen module paths**: Initial code referenced `bindings::*` but macro emits `generated_module::interface0::*` namespaces
   - Fixed by using macro-generated paths directly in on_request/on_response methods
2. **Component instantiate return type**: Expected tuple unpacking but wasmtime 31 returns the type directly
   - Fixed by removing tuple destructuring and using returned type as-is
3. **Pooling allocator usize conversion**: `max_component_instance_size` expects usize, not u64
   - Fixed with `try_into().map_err(...)` to handle conversion with error propagation

### Debug Derive Constraint (WasmRuntime, Component)
- **Issue**: WasmRuntime and Component don't implement Debug; RuntimeHandle and WasmFilter couldn't derive Debug
- **Fix**: Removed Debug derives from WasmFilter and RuntimeHandle; left Clone (sufficient for Arc-wrapped use)

## Open Questions & Deferred Work

### Wallclock Interruption Not Yet Integrated
- WIT defines filter execution model but no hard timeout enforcement in host
- Wasmtime supports fuel-based interruption; not yet wired to config or observability
- Deferred to v0.4 hardening; filters can exceed budget without penalty in v0.3

### Instance Reuse Optimization Tuning Deferred
- Pooling allocator pre-allocates instances; utilization and cache effects not yet measured
- Benchmarks needed to validate that pooling amortizes cold-start cost across throughput
- Per-request instantiation would exceed hot-path budget; pooling is required for production
- Tuning parameters (pool size, memory pre-allocation) require production workload data

### Streaming Response Filters (v2 ABI) Not in Scope
- Current `on-response` fires after body fully buffered; filter cannot intercept stream chunks
- v2 WebAssembly Component Model with streams/resources would enable response filtering
- Deferred pending upstream wasmtime evolution; v0.3 covers request and buffered response

### Binary Release Hardening
- Single backend routing still in place (KvAwareRouter → HedgedRouter → CircuitBreakerArbiter → BackendId(0))
- Load balancing across multiple upstreams not yet wired to CLI config
- Deferred; binary is operational for single-backend proxy use cases
