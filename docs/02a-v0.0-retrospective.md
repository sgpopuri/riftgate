## What went well

- **Build-in-public discipline held end-to-end.** The repo went from empty to a fully wired-up design surface in one milestone, with every load-bearing decision traceable to an Options doc plus an ADR. This is rare and worth saying so.
- **Mid-flight naming refinement was the system working as intended.** The renaming from `PerCoreScheduler` (the LLD's earlier outline-stage term) to `PerShardScheduler` ([ADR `0004`](06-adrs/0004-per-shard-default-stealing-opt-in.md)'s final term) was a clarification that the `v0.1` reality is M logical shards on N Tokio threads with no pinning — physical thread-per-core is a `v0.2`-and-later deployment-shape question, not a scheduler-trait question. Catching this in design rather than in code is the documentation-first pillar paying for itself.
- **The 2026-05 research pass was absorbed via the project's own discipline.** When two real gaps surfaced (in-proc rate limiting trait + MCP capability brokering), they were absorbed as Options docs (`021`, `026`), LLDs (`lld-rate-limiter`, `lld-mcp-capability`), and proposed ADRs (`0009`, `0015`) — not as ad-hoc additions to the existing files. The system handled the new scope.
- **Honest scoping in the vision.** [`docs/00-vision.md` §4 and §8](00-vision.md) explicitly name what we declined to ship — multi-provider adapters as a kernel feature, a semantic-cache reference impl, distributed state substrate. Stating non-goals up front is the kind of move that pre-empts category-of-criticism a year out.

## What we missed (and fixed in the close-out PR)

This retrospective is the surface that records what we did not catch during the milestone itself. The following items were surfaced during the close-out audit and corrected before opening `v0.1`:

- **Chapter-citation leakage from a private curriculum into the public repo.** Throughout the original `v0.0` corpus (Options docs, LLDs, HLD, plane narratives, ADRs, roadmap), the project cited a private source-systems curriculum by chapter number (`Ch1`, `Ch3`, `Ch16`, etc.) and by sibling-folder file path (`hashing/ch07_consistent_hashing.md`, `trees/ch04_heaps_priority_queues.md`, `advanced/ch08_design_data_structures.md`, `graphs/ch03_topological_sort_dags.md`, etc.). External readers of the GitHub repo had no access to that material. The close-out flipped the convention: every citation now names the technology, pattern, paper, or named system directly (`io_uring`'s shared SQ/CQ ring model, ARIES-style write-ahead logging, work-stealing schedulers, etc.), and every Options doc carries a `## References` section listing concrete external sources (RFCs, kernel.org docs, papers, source repositories, books). [`AGENTS.md` §11](../AGENTS.md) records the convention.
- **A misattribution caught by the rewrite.** Several files cited "`Ch8` (backpressure as policy)" — but in the underlying curriculum, `ch08` is pub/sub messaging, not backpressure. The rewrite incidentally fixed this drift: the relevant prose now refers to "backpressure as policy" by name, with citations to Hellerstein and the broader flow-control literature.
- **Count drift across project surfaces.** [`README.md`](../README.md), [`AGENTS.md` §2](../AGENTS.md), and the [v0.0 deliverables block in the roadmap](02-mvp-roadmap.md) all said "eight LLDs" and "eight Options docs / eight ADRs" — but the filesystem at tag time has ten LLDs (eight core plus the two research-pass), nine Options docs (the seven foundational from the original octet plus `021`, `026`), eight accepted ADRs, and two `proposed` ADRs. The close-out reconciled the wording with the filesystem.
- **"Three planes" wording vs the four-plane architecture.** [`hld.md` §1](03-architecture/hld.md), the v0.0 deliverables block in the roadmap, and one line in [`control-plane.md`](03-architecture/control-plane.md) all referred to "three planes" — but the architecture is four (data, control, extension, observability). The rest of the HLD and the four plane-narrative files were already consistent with four. The close-out unified the wording.
- **No markdown bodies for the two research-pass ADRs.** ADRs `0009` and `0015` were originally indexed as "reserved (proposed)" in [`docs/06-adrs/README.md`](06-adrs/README.md) without corresponding `0009-*.md` and `0015-*.md` files. The close-out wrote both files in `proposed` status with full Michael-Nygard structure, and updated the index to point at them.

## Process notes for `v0.1` and beyond

- **Every milestone gets its own retrospective doc** at close, anchored to the milestone's git tag. This file is `v0.0`'s; `v0.1`'s will be `02b-v0.1-retrospective.md`, and so on. The retrospective is a permanent artifact: written once, never edited.
- **The "Currently shipping" block at the top of [`docs/02-mvp-roadmap.md`](02-mvp-roadmap.md) is the live-state surface, not a history surface.** It tracks what is in flight and what was just shipped; once a milestone closes, the close-out narrative moves out of "Currently shipping" and into the corresponding retrospective.
- **The `v0.0` close-out establishes the citation discipline that every subsequent milestone follows.** No chapter numbers from any private curriculum appear in the repo; every Options doc carries a `## References` section listing concrete external sources. The convention is enforced by [`AGENTS.md` §11](../AGENTS.md) and lives in the Options-doc template.
- **No code lands in `v0.1` without a justifying ADR.** This is the discipline already encoded in [`AGENTS.md` §5](../AGENTS.md); calling it out here so the `v0.1` retrospective can cite back to it.
